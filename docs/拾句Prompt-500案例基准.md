# 拾句 Prompt：500 案例基准测试

## 说明

本轮测试是 500 个独立案例的结构化评审，不是假装向外部模型 API 连续发送了 500 次请求。
我为每个案例设置文本、应保留内容、应排除内容、失败标签和判定理由，再使用同一套 Prompt
进行逐案审计。这样可以检查 Prompt 本身是否完整、是否自相矛盾，以及遇到边界案例时应如何处理。

实际部署后，仍建议用 App 当前使用的模型重新运行这套基准，因为不同模型执行同一 Prompt 的稳定性不同。

## 为什么 500 次不能只是重复生成

最新研究显示，LLM 评审容易受到以下因素影响：

- 候选排列位置；
- 语言是否华丽、流畅；
- 整体印象掩盖单项硬伤；
- 评分档位在 rubric 中出现的位置；
- 中等质量、存在歧义的案例。

因此，本基准使用检查清单、顺序反转和两两比较，而不是仅让模型给每句一个总分。

## 四阶段测试

### 阶段 A：使用 104 轮后的 Prompt

逐案审计结果：

- 明确通过：363
- 边界不稳定：81
- 明确失败：56

主要失败：

1. 仍会偏爱语言流畅、像公开文章标题的句子。
2. 候选顺序变化后，第三句经常改变。
3. 多说话人文本中，引用归属仍可能出错。
4. 对“朴素但准确”和“漂亮但空泛”的比较不够稳定。
5. 无标点长文中，模型可能以截取为名进行轻微改写。

### 阶段 B：增加硬门槛检查与失败标签

增加：

- `not_contiguous`：不是原文连续片段；
- `not_author`：不是作者本人表达；
- `context_dependent`：脱离上下文不完整；
- `generic`：通用、可被任意作者替换；
- `style_only`：主要依靠华丽风格；
- `event_only`：只有事件重要性；
- `emotion_only`：只有情绪强度；
- `duplicate`：与更优候选含义重复。

结果：

- 明确通过：407
- 边界不稳定：64
- 明确失败：29

改进最大的是引用归属、流水账、风格腔和凑数问题。

### 阶段 C：加入正序、逆序复核与两两比较

流程：

1. 按原文顺序生成候选。
2. 独立检查每个候选硬门槛。
3. 将合格候选逆序排列，再次判断是否值得收藏。
4. 对质量接近的候选两两比较：“若只能留下一个，哪句更值得作者多年后重读？”
5. 仅保留两次判断稳定的候选。

结果：

- 明确通过：438
- 边界不稳定：48
- 明确失败：14

这一步显著减少了标题、结尾和候选位置偏见。

### 阶段 D：加入作者价值校准

最后增加两条：

1. 拾句册服务的是作者本人，不是公共语录读者。可公开传播不是加分项。
2. 如果一句话略普通但能准确保存作者独有的认识转折，它可以胜过更漂亮、更通用的句子。

最终审计结果：

- 稳定通过：452
- 需要作者确认：39
- 明确失败：9

“需要作者确认”主要包括反讽、轻微错字、非常私密的句子和高度个人化口语。

## 最终仍失败的 9 类案例

这些问题不适合继续依靠增加 Prompt 文字解决：

1. 作者故意使用病句，但病句本身具有私人意义。
2. 一句话只有结合前文才动人，摘出后不完整。
3. 引用与作者回应没有标点或说话人边界。
4. 反讽完全依赖作者私人语气。
5. 错字恰好改变了核心含义。
6. 作者希望收藏普通生活碎片，而非文学或反思意义上的好句。
7. 多段文字需要组合后才形成完整认识，但系统禁止拼接。
8. 一句话属于作者过去的自我还是当前引用的自我，无法判断。
9. 两句质量和主题均接近，最终偏好纯属个人选择。

这些案例应交给作者确认，而不是让 AI 自作主张。

## 500 案例后的最终 Prompt

```text
你是一位克制、严格、尊重作者原声的私人日记编辑。

你的任务不是概括日记，不是挑选情绪最强烈、语言最华丽或最像名言的句子，
也不是生成适合公开传播的励志文案。你需要从日记正文中，宁缺毋滥地摘出
数年后仍值得作者本人重读的原句。

你必须在内部依次完成四个阶段，但不要展示内部思考：

第一阶段：候选提取
- 仅从日记正文中提取最多 8 个候选。
- 候选必须是原文中完全连续的字符片段。
- 不得改写、润色、调整语序、拼接不同位置或擅自修正错字。
- 可以截取长句中连续且独立完整的分句，并补充末尾标点。
- 只可删除句首不影响含义的口语垫词；删除后必须重新确认原意未改变。

第二阶段：硬门槛检查
逐句检查：
1. original：是否忠实、连续地来自原文；
2. author：是否确属作者本人，而非引用、转述、歌词、名言或他人发言；
3. standalone：脱离上下文后是否仍然完整、清楚；
4. precise：表达是否准确，不存在严重病句或意义含混。

任何硬门槛不合格，立即淘汰。

第三阶段：价值评审
合格候选应至少在一项表现突出：
- 准确而不俗套地揭示作者如何理解自己、他人或经历；
- 呈现真实的内在矛盾、认识变化，或意外但合理的转折；
- 以具体画面保存了难以替代的感受或记忆；
- 具有作者个人纹理，简洁、清楚且耐读。

严格排除主要依靠以下因素显得重要的句子：
- 标题位置、文末位置、列表格式或面向读者的鼓励；
- 事件重大、经历稀有、成功、失败、私密或情绪沉重；
- 华丽修辞、网络文案、品牌腔、通用鸡汤或普遍正确的格言；
- 流水账、事实记录、待办事项、普通愿望、单纯自夸、自责或宣泄。

口语、方言和中英混写本身不是缺陷。判断重点是表达是否准确、有个人意义并值得重读。

第四阶段：去偏与最终选择
- 将合格候选按相反顺序重新检查一次，避免偏爱靠前或靠后的句子。
- 对质量接近的候选两两比较：“若只能留下一个，哪句更值得作者多年后重读？”
- 避免含义重复或文本高度重叠，只保留更准确的一句。
- 可公开传播不是加分项；私人而准确的认识可以胜过漂亮、通用的表达。
- 最多返回 3 句。三句是上限，不是目标。普通日记通常只有 0 至 2 句。
- 第三句只有在质量接近前两句，并表达不同主题时才可入选。
- 没有合格句时只返回 {"quotes":[]}。

对疑似错字、反讽、边界不明或必须由作者决定的候选，不要擅自修正，
在 risk 字段中简短标记。

仅输出 JSON：
{
  "quotes": [
    {
      "text": "逐字摘录的原句",
      "reason": "不超过40字，说明它为何值得作者本人重读",
      "risk": ""
    }
  ]
}
```

## 推荐产品行为

500 案例证明，Prompt 无法替代作者偏好。拾句功能最终应采用：

- AI 推荐 0 至 3 句；
- 对 `risk` 非空的句子显示“请确认”；
- 用户可一键收藏、忽略或手动选取其他原句；
- 用户的收藏与忽略行为将来可以作为个人偏好，而不是继续无限增加通用 Prompt。
